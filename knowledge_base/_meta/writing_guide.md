# 文档写作规范

> **适用范围**：knowledge_base/ 下所有 115 篇文档  
> **版本**：v1.0  
> **创建日期**：2026-05-11

---

## 一、写作风格

### 1.1 人称与语气
- 使用**第三人称客观叙述**，不用"我们""您"
- 例外：常见问答章节中的"问题"部分可用第一人称模拟用户口吻
- 语气：专业但不生硬，像产品手册而非学术论文
- 禁止营销话术（"震撼来袭""极致体验"）

### 1.2 长度范围
| 文档类型 | 目标字数 | 允许浮动 |
|---|---|---|
| 商品详情页 | 1500 字 | 1200-1800 |
| 选购指南 | 1800 字 | 1500-2200 |
| 产品使用手册 | 2500 字 | 2000-3000 |
| APP 操作手册 | 1500 字 | 1200-1800 |
| 配网/联动指南 | 1500 字 | 1200-1800 |
| 售后政策 | 1200 字 | 1000-1500 |
| 故障排查 | 2000 字 | 1500-2500 |
| 错误码手册 | 2000 字 | 1500-2500 |

### 1.3 格式规范
- 标题层级：最多用到三级（`#` / `##` / `###`）
- 参数表、对比表等必须使用 Markdown 表格
- 操作步骤使用有序列表（1. 2. 3.）
- 并列项使用无序列表（-）
- 不使用 emoji
- 不使用 HTML 标签

## 二、数据真实性规则

### 2.1 核心原则
所有商品均为小米/Redmi/米家/生态链（石头等）的真实产品，参数从官方渠道获取，不做虚构。

### 2.2 分类标准

| 数据类型 | 处理方式 | 示例 |
|---|---|---|
| **商品名/型号** | 使用真实产品名，按 product_mapping.md 映射 | "石头扫地机器人 T7""小米 14 Pro" |
| **Doc ID** | 按文档清单中的编号，不可修改 | "PROD_PHONE_001" |
| **关联意图** | 按文档清单中的映射，不可修改 | "S2/S3/S6" |
| **商品参数** | 从官网/评测获取真实数据 | 处理器型号、屏幕尺寸、吸力值 |
| **价格** | 使用发布价或当前售价 | "2799 元" |
| **配件型号** | 使用真实型号和兼容关系 | 边刷 T7/T7S 通用 |
| **操作步骤** | 参考真实产品说明书和 APP 界面 | 米家 APP 实际操作流程 |
| **错误码** | 使用真实错误码体系 | 石头 T7 的 E1-E20 |
| **售后政策** | 平台政策为虚构（比特严选），需自洽 | 保修期、退换条件 |

### 2.3 数据来源优先级
1. 官方产品页（mi.com / roborock.com）
2. 官方说明书 PDF
3. 权威评测（ZOL、中关村在线、什么值得买）
4. 用户社区验证的信息

### 2.4 自洽性要求
- 同一商品在不同文档中出现时，参数必须一致
- 配件兼容关系必须双向一致（A 兼容 B ↔ B 兼容 A）
- 产品代际关系需真实（T7 → T7S Plus 的升级点要准确）
- 售后政策虽为虚构，但同一政策在不同文档中引用时数字必须一致

## 三、与意图体系的映射

### 3.1 映射原则

每篇文档在 metadata 的 `related_intents` 中标注它能服务的意图。映射来自文档清单，不可自行添加或删除。

### 3.2 各意图对文档内容的要求

| 意图 | 对文档内容的具体要求 |
|---|---|
| S1 选购推荐 | 选购指南中必须有明确的推荐型号和理由 |
| S2 参数咨询 | 商品详情的参数表必须字段完整、结构化 |
| S3 对比选购 | 同品类商品的参数表字段必须对齐（可跨文档比较） |
| S4 价格活动 | 价保政策中需有明确的价格保护规则 |
| S6 配件兼容 | 商品详情的配件兼容章节必须列出具体型号 |
| S7 适用场景 | 商品详情和选购指南中的场景描述必须具体 |
| S8 操作指引 | 使用手册的操作步骤必须逐步编号、可执行 |
| S9 配网连接 | 配网指南必须写明环境要求和常见失败原因 |
| S10 APP 功能 | APP 手册必须写明功能入口的完整导航路径 |
| S11 固件升级 | 错误码手册中需覆盖升级失败的处理 |
| S12 生态联动 | 联动指南必须写明支持的设备组合 |
| S13 保养维护 | 使用手册的保养章节需有具体周期和方法 |
| S14 售后政策 | 保修政策必须有明确的品类覆盖和期限 |
| S15 退换货 | 退换政策必须有明确的条件和流程 |
| S16 物流配送 | 物流政策必须有时效和范围 |
| S17 发票/会员 | 发票会员政策必须有具体规则 |
| F1 故障报告 | 故障树必须有决策树结构和转人工条件 |

### 3.3 跨文档引用规则
- 同一信息只在一个文档中完整出现
- 其他文档用引用格式指向源文档：`详见 [{doc_id}] {文档标题}`
- 引用格式示例：`详见 [MANUAL_VAC_003] 扫地机保养维护手册`

## 四、RAG 检索友好性

### 4.1 章节标题
- 标题应包含关键词，便于检索匹配
- 好的标题："详细参数""配件兼容""退换货条件"
- 差的标题："更多信息""其他""补充说明"

### 4.2 表格设计
- 参数表的列名统一（"类别 / 参数项 / 数值"）
- 同品类商品的参数表字段保持对齐，支撑跨文档对比

### 4.3 chunk 切分友好
- 每个二级标题（`##`）下的内容应当语义自足
- 避免在表格中间换章节
- 章节内容不宜过长（单章节不超过 500 字）
